# Towards Evaluation of Implicit Software World Models in Coding LLMs

> Source: arXiv:2606.27406v1 (full HTML, fetched 2026-06-29). Authors: Egor Bogomolov, Yaroslav Zharov (JetBrains Research). DL4Code @ ICML 2026. Data: huggingface.co/JetBrains-Research; Code: github.com/JetBrains-Research/cwm-execution-tracer.

## Abstract

Software engineering, whether performed by humans or by AI agents, requires reasoning about how software behaves. We call the internal model that supports such reasoning the *software world model*, and view current code-execution benchmarks as covering one well-studied slice of it — control flow. We shift the observable axis to execution resources: alongside test outcome and exception class, we predict peak memory, wall-clock time, and ranked profiler outputs at method and line granularity. We use SWE-bench Verified as the data source. All tested models, frontier included, show modest performance and brittle behaviour, suggesting a lack of understanding of how software is executed vs how source is written.

## 1 Introduction

Function-level benchmarks (HumanEval, MBPP) score pass on held-out tests; repo-level (SWE-Bench, Aider-Polyglot) score whether a patch fixes a bug; repo-scale (Commit0) reproduces a repo from spec. These measure code-generation capability. A second line evaluates execution understanding: CRUXEval (input/output prediction on 800 functions), REval (coverage, state, next-statement, output), ThrowBench (exception type), BigO(Bench) (asymptotic complexity), CodeMind (1,450 programs), RE2-Bench (repo-scale). These mostly use isolated synthetic functions, focused on return value. Control flow is one slice; the broader *software world model* covers build/dependency resolution, test/CI behaviour, runtime, concurrency, repo-level patch interaction. Implicit (spontaneous from code corpora) vs explicit (trained, e.g. CWM). This paper stays in code execution but pushes to realistic contexts; two gaps: (1) function snippets ≠ practical software, (2) return value is one facet. They (1) design metrics for wider execution behaviour, (2) collect library-level cases from SWE-bench Verified, (3) test recent code-fluent LLMs.

## 2 Data

SWE-bench Verified: 500 issues, verified gold patches, 12 Python repos. Samples = test pairs failing→fixed. Keep <500K chars context; downsample Django-heavy to 435 examples, repo diversity preserved. Inject sys.settrace/sys.monitoring tracer in Docker, run before+after patch. Observables: test outcome (pass/fail + exception class); wall-clock (clean run, no trace); peak memory (line-level max); profiler (method+line × time+memory, top 20 rows, repo scope).

## 3 Metrics

Outcome = classification accuracy/precision/recall/F1. Resource = linear calibration on log10 (slope a, bias b, MAE; ideal 1,0). Zero predictions → 0.01ms/10KB. Profiler = ranking: Recall@5, NDCG@5; execution rate = fraction predicted names actually executed.

## 4 Experiment setup

Models: Anthropic claude-haiku-4-5, sonnet-4-6, opus-4-7; OpenAI gpt-5-mini, 5.2, 5.4, 5.5; open-weight gpt-oss-120b, Qwen3.5-397B-A17B, Qwen3-235B-A22B, Qwen3-30B-A3B, FAIR CWM. Context = executed library slice + 60K test file window. Output 10-field JSON (reasoning, outcome, failure_line, exception_type, peak_bytes, wall_ms, hot_methods/lines × time/alloc).

## 6-7 Limitations/Conclusion

Single oracle context, SWE-bench-only, Python-only. First probe; envision build/CI/deploy/concurrency/agentic extensions.

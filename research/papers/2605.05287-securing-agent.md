# Securing the Agent: Vendor-Neutral, Multitenant Enterprise Retrieval and Tool Use

> Source: arXiv:2605.05287v1 (6 May 2026, full HTML). Arceo & Narsing, Red Hat AI. ACM CAIS '26. Impl: OGX (open, OpenAI-compatible). CC BY 4.0.

## Core gap

Retrieval ranks by relevance, not authorization → a tenant query can surface another tenant's confidential doc that scores highest. Ungated RAG leaks cross-tenant in 98–100% of probes. Secure retrieval = {d: relevance(q,d)>θ ∧ P(u,d)=permit}.

## Architecture

- 3 data layers: (1) policy-aware ingestion (tenant metadata at ingest); (2) retrieval gating (ABAC pre-search + chunk filter, predicate pushdown); (3) shared inference (cost O(N·M)→O(M)). Server-side orchestration = control path; client controls *what*, not *how*.
- Threat model: malicious tenant, compromised client, buggy tool; out-of-scope: server compromise. A3: inference untrusted — gate before context.

## Results

ABAC gating → CTLR/AVR = 0% both orchestration modes; +19ms; precision 2.2×; 90/90 injection blocked under gating (62–80% leak ungated). No-pushdown recall degrades 1.0→0.002 @50K chunks. Client ~2× QPS but trust depends on client; server-side = enforcement guarantee.

## Use

Ch5 governance/Ch4 RAG: relevance≠authorization; gate at retrieval; server-side enforcement.

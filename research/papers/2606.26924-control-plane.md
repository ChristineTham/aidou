# A Deterministic Control Plane for LLM Coding Agents

> Source: arXiv:2606.26924v1 (25 Jun 2026, full HTML fetched 2026-06-29). Padmaraj Madatha, Happiest Minds (AIP CoE). Reference impl: Rel(AI)Build. Zenodo 10.5281/zenodo.20780913.

## Abstract / thesis

LLM coding harnesses grant broad file/shell access, but the config layer (rules files, agent defs, IDE markdown) is unmanaged. Prevalence study of 10,008 repos (6,145 agent configs): **10.1% fork-adjusted exact-duplicate config paths**; **75.5% clone pairs cross-org**; 58% single-commit; <1% declare permission boundaries. Proposes a deterministic control plane *above* the harness, mapping 1:1 to five gaps. Determinism is load-bearing — every control is testable code, not an LLM.

## Five structural gaps → mechanisms

1. Config is unmanaged software → SHA-256 content addressing, HMAC lockfile, hash-chained audit log, SBOM (NTIA/EO14028/SLSA L2).
2. Execution unbounded → phase state machine, 3-iteration cap, requirement→file→test trace.
3. Expertise doesn't port → define-once compile-to-7 IDE targets.
4. Unbounded exec w/o traceability; 5. expertise locked to dialect.

## Mechanisms

- Five resource pillars: agents/skills/knowledge/profiles/workflows. 5 permission tiers (readonly→orchestrator), fail-closed. Attack-derived blocklist (Codecov pipe-to-sh, ua-parser detached-spawn, xz persistence, git hooks). Jaccard drift (LOW≥0.8, HIGH<0.6). 8-phase Cooper stage-gate, 4 HITL gates. Threat model T1-T7 (prompt injection Medium residual).
- Conformance: 237 defs ×7 targets; 10 tamper/15 over-priv/20 blocklist all caught — enforcement not outcome.
